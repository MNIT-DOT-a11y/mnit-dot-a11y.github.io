API Contracts & Accessibility

Strong API contracts help prevent accessibility defects by making required relationships explicit instead of forcing developers to guess.

Scenario: A User Profile Card

On the left, a vague model forces guessing and alt text is forgotten. On the right, a clear model makes the data relationships obvious.

Not Accessible: Vague Data Model

<div class="product-card">
  <h3>}} product.name }}</h3>
  <img
    [src]="product.imageUrl"
    [alt]="product.imageAltText">
</div>

Accessible: Clear, Descriptive Contract

export interface UserProfile {
  displayName: string;
  jobTitle: string;

  // Naming and grouping makes the relationship obvious
  profileImageUrl: string;
  profileImageAltText: string;
}

Scenario: Displaying a Product Image

If an API only sends an image URL, the front-end developer is forced to either omit alt text (an accessibility failure) or guess with something unhelpful like “product image”.

This places responsibility for accessibility on the wrong party. The API should provide the required information.

JSON from API

{
  "id": "cfa...",
  "name": "Heavy-Duty Work Gloves",
  "imageUrl": "/images/gloves.jpg",
  "imageAltText": "A pair of durable, brown leather work gloves."
}

TypeScript Interface

export interface Product {
  id: string;
  name: string;
  imageUrl: string;
  imageAltText: string;
}

Angular Component Template

Inaccessible
<div class="product-card">
  <h3>}} product.name }}</h3>
  <img [src]="product.imageUrl" alt="Product Image">
</div>
Accessible
<div class="product-card">
  <h3>}} product.name }}</h3>
  <img
    [src]="product.imageUrl"
    [alt]="product.imageAltText">
</div>

The accessible version uses property binding: [alt]="product.imageAltText". This ensures specific, descriptive alt text is always present.
